# natural-japanese

日本語の文章から「AI臭さ」を取り除き、自然な文章に仕上げるスキル。対象は note・ブログ・エッセイ・技術文書など全般。

## 設計思想

AI は自分自身の AI 臭さを認識しにくい。だから修正の前に、まず機械で決定的に検出する。検出後にその指摘をどう直すかは AI（あなた）が判断する。「検出は機械、判断は人間（またはAI）」という役割分担がこのスキルの軸になる。

## 0. まなざし（文体プロファイル）— 任意

`style-profile.md`（プロジェクトルートかユーザー指定の場所）が既にあれば、執筆・リライトの前に読み込み、そこに書かれた視点・語彙・リズムの癖を下敷きにする。

なければ**汎用モードで進めてよい**。プロファイルは必須ではなく、AI臭除去そのものは汎用モードでも機能する。

ユーザーが「自分の文体を学ばせたい」「プロファイルを作りたい」と求めた場合のみ、セットアップフローに入る。

### セットアップフロー（求められたときだけ）

1. ユーザーの過去文章を3〜5本読む（note・ブログ・過去原稿など、ユーザーに場所を尋ねる）
2. `assets/style-profile-template.md` の観察項目（視点・文末の癖・リズム・語彙・構成）に沿って特徴を抽出する
3. テンプレートを埋めた `style-profile.md` を書き出す。断定しすぎず、「傾向として」と留保をつけて書く（3〜5本という少ない標本で言い切ると外れやすい）

## 1. 執筆前の設計 — 揺らぎ

下書きに入る前に、扱うトピックの重要度に意図的なムラをつける。すべての段落を同じ熱量・同じ厚みで書くと、それ自体が「整いすぎた不自然さ」になる。詳しい手順は `references/revision-guide.md` の「揺らぎ設計」を参照。

### 素材集め（再帰的推論リサーチ）— 任意、新規執筆時

新規に書き起こす場合、揺らぎ設計の前後で素材（実例・数値・一次情報）を集める工程を挟むと後が楽になる。手元に固有名詞や数字が乏しいまま書き始めると、後段の判断台帳で「一般論しか言えていない」と気づいても直しようがない。具体的な手順は `references/revision-guide.md` の「素材集め」を参照。推論と検索の往復のやり方、十分と判断する基準、Web検索不可の環境でのユーザーへの素材提供依頼をまとめてある。

## 2. 下書き

プロファイル（あれば）と揺らぎ設計を踏まえて書く。この段階では禁止語やリズムを気にしすぎず、まず内容を出し切ってよい。

## 3. 静的検知

```
uv run scripts/ai-smell-lint.py --json <file>
```

このスクリプトは禁止語・翻訳調・否定肯定対比の反復・文長の均質さ・体言止め率・段落頭の接続詞率・語彙多様性・英語統語の疑いなどを機械的に検出する。検出結果は件数に関わらず exit code 0（CI ゲートではなく lint）。ファイル不在・ディレクトリ指定・読み取り不可などの入力エラーのときだけ exit code 1。

`uv` が使えない環境（Claude.ai 等）では、代わりに `references/manual-checklist.md` の文章版チェックリストで同じ観点を人手でなぞる。

収束駆動ループ（4〜6）を回すときは、直前の `--json` 出力を `--baseline` に渡すと、今回の結果を resolved（解消）/ new（新規）/ persisting（継続）に自動で仕分けてくれる（`uv run scripts/ai-smell-lint.py --json --baseline <前回の.json> <file>`）。

## 4. 検出結果の判断 — 判断台帳

lint の `findings` はあくまで疑いの提示であり、機械的に全部直せという指示ではない。直す前に一手間かける。今回ヒットしたカテゴリの節を `references/revision-guide.md` で読み直す（理由は同ファイル冒頭を参照）。そのうえで対応表を確認し、文脈に照らして「直す/直さない」を判断する。

段落が一般論しか言えていない（固有名・数値・実例がない）場合は、書き方でなく素材の問題であることが多い。修正ループをそのまま回し続けず、`references/revision-guide.md` の「素材不足の分岐」を見て情報収集に戻るべきか判断する。

判断はその場で流さず、finding 一つひとつに対して「直した」か「残す（理由）」かをその場で書き残しながら進める。台帳の形式・記入例は `references/revision-guide.md` の「判断台帳」を参照。

用語カタログが必要なら:
- 禁止語・LLM常套句の詳細と理由 → `references/forbidden-patterns.md`
- 英語統語・翻訳調のパターンと書き直し例 → `references/translationese.md`

lint のカテゴリに当てはまらない「そもそも読みにくい」文章に見えるときは、`references/readability-principles.md`（語順・読点・一文一義・パラグラフなど、ジャンル横断の一般原則）と `references/readability-antipatterns.md`（人間の書き手もやりがちな悪文カタログ、lint 未対応の項目は目視確認と明記）を参照する。ジャンル（tech/business/essay/公用文）によって原則の重みづけが変わる場合は `references/genre-notes.md` の差分を当てる。

## 5. 再検知・収束

台帳の「直した」項目を反映したら lint を再実行し、新しい finding が出ていないか確認する。台帳上の全 finding が「直した」か「残す（理由）」に仕分けられ、かつ「直した」による修正が新たな finding を生んでいない状態になるまで 3〜4 を繰り返す。同じ finding が2周連続で直しては再発する場合の扱いは `references/revision-guide.md` の「発散ガード」を参照。

## 6. 最終パス — 自己点検ループ

lint が収束しても、それは既知のAI臭パターンが表層から消えたことしか意味しない。最後に必ず、初見の人間として通読し、声に出して読むつもりでリズムを確かめる最終パスを行う。手順は `references/revision-guide.md` の「自己点検ループ」を参照。ここで違和感を見つけたら、それを台帳に新しい項目として起こし、5に戻る。この最終パスで違和感がなくなったら完了とする。

## 7. 後片付け

完了したら、作業中に作った中間ファイル（台帳ファイル・lint の JSON 出力・下書きのバックアップ等）をすべて削除する。ユーザーのプロジェクトに残してよいのは完成した文書と、ユーザーが明示的に望んだ場合の `style-profile.md` だけ。中間ファイルの置き場所と削除の詳細は `references/revision-guide.md` の「作業ファイルの扱い」を参照。
