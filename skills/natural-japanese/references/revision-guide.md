# 自己点検ループと揺らぎ設計

## なぜ周回ごとに読み直すのか

一度通しで読んだだけで修正を続けると、基準が少しずつ緩む。読んだ直後は「理解した」という感覚が残る。だが実際に手を動かしているのは原文でなく記憶の要約だ。要約は元の基準より粗い。例外条件や境界線がそこで抜け落ち、周回を重ねるたびに書き手の都合のいい方向へ寄っていく。

打ち手は単純で、SKILL.md の§4のとおり、その周でヒットしたカテゴリに対応する節をそのつど読み直す。内面化は一度の理解では終わらない。同じ場所に何度も戻ることで少しずつ進む。参考記事（[note.com/art_reflection](https://note.com/art_reflection/n/ncdb00d92bbde)）も同じ点を指摘している。基準を読んだと申告しながら出力で基準を破る現象は、要約記憶だけで作業しているときによく起きる。

## 揺らぎ設計（執筆前）

AIが書く文章は、すべての段落を同じ熱量・同じ厚みで処理しがちになる。話題の重要度に関わらず均等に力を配分するので、結果として全体が「よくできているが誰の実感もこもっていない」文章になる。人間の書き手は違う。強く語りたい箇所は長く粘り、どうでもいい経緯は一言で流す。この濃淡そのものが自然さの正体に近い。

下書きに入る前に、扱う論点を軽くリストアップし、それぞれに「どれだけ語る価値があるか」を仮に割り振っておくとよい。すべてが同格に見える構成案ができあがったら、それは執筆前の段階で既に危険信号だと思ったほうがいい。

濃淡をつける具体的な手がかり:

- 一番言いたい主張はどれか。それ以外は説明の补助でしかないと割り切る
- 自分が実際に迷った・失敗した箇所はどこか。そこは具体的に書く価値がある
- 逆に、前提として片付けてよい話は1文で済ませる
- 段落の長さを事前に決めない。内容が短く済むなら短く終わってよい

これは「短くする」ことを目的にした技法ではない。重要な箇所により多くの言葉を使い、そうでない箇所からは思い切って言葉を引き上げることが目的で、結果として文章全体の輪郭にメリハリが出る。

## 素材集め（再帰的推論リサーチ）

揺らぎ設計で厚く書く箇所を決めても、固有名詞・数値・一次情報が手元になければ、その段落は結局一般論で埋まる。新規に書き起こすときは、揺らぎ設計の前後で素材集めの工程を挟むとよい。

やり方は推論と検索の往復にする。まず自分の理解で仮の主張と仮の根拠を組み立てる（推論）。その根拠を検索で裏づけるか崩す（検索）。出てきた情報をもとに主張を組み直す（再推論）。これを1〜2周する。検索だけを機械的に繰り返すのは避けたい。何を探しているかが曖昧なままだと、的外れな情報ばかり集まる。推論だけで済ませるのも避けたい。裏づけのない断定になる。

何を集めるかの目安:

- 固有名詞。製品名・人名・地名・サービス名など、一般名詞に置き換えると成立しなくなる語
- 数値・日付。できれば出典つきで。概数しか出せない場合はその旨を書く
- 一次情報。本人の発言・公式ドキュメント・実際に起きた出来事。伝聞よりも一次に近いものを優先する

十分と判断する基準はひとつ。一番言いたい主張を、固有名詞・数値・具体的な出来事のどれも使わずに説明できるかを自問する。説明できてしまうなら、その主張はまだ一般論の域を出ておらず、素材が足りていない。

Web検索が使えない環境（Claude.aiなど）では、この往復を自力で完結できない。検索を省いて推論だけで書き進めるのは避け、ユーザーに「この主張を裏づける具体例・数値・一次情報はあるか」と素材の提供を求める。ユーザーが素材を持っていないこともある。提供する気がないこともある。そのときは内容が汎用的になる旨を断って、そのまま進めてよい。

## lint 出力（カテゴリ）ごとの対応方針

| カテゴリ | 何を疑うか | 修正の方向 |
|---|---|---|
| `forbidden_phrase` | LLM常套句・結論の押し付け | `forbidden-patterns.md` の該当カテゴリを読み、その表現が本当に必要か検討する。多くは削るか、より具体的な言い方に置き換えられる |
| `translationese` / `translationese_morph` | 「〜することができる」等の英語直訳調 | `translationese.md` の対応パターンを見て、before/after の型に沿って書き直す |
| `antithesis_repetition` | 「〜ではなく」型の対比が3回以上 | 全部直す必要はない。最も重要な対比だけ残し、他は素直な肯定文・具体例に変える |
| `low_sentence_variance` / `low_burstiness` | 文長が均質でリズムが単調 | 一番言いたい文を思い切って短くする、または背景説明の文をあえて長く続ける。短文と長文を意図的に隣接させる |
| `high_length_autocorrelation` | 文長パターンが固定的に繰り返す | 直前の文と似た長さの文が続いていないか確認し、意図的に長さを崩す |
| `nominal_ending` | 体言止めが多すぎる | すべてを用言止めに戻す必要はないが、連続する体言止めのうち1つおきくらいは通常の文末に戻す |
| `paragraph_lead_conjunction` | 段落頭の接続詞が多い | 接続詞なしで前の段落とつながるように書き直せないか考える。無理なら残してよい |
| `uniform_paragraph_structure` | 段落あたり文数が揃いすぎ | 揺らぎ設計に立ち返り、重要な段落を長く、そうでない段落を短くする |
| `repeated_sentence_lead` / `repeated_syntax_template` | 文頭の型・構文テンプレートの使い回し | 冒頭の言い回しを変えるか、その文自体を別の構造（疑問形・体言止め・引用から始める等）に組み替える |
| `low_lexical_diversity_ttr` / `_mtld` | 語彙の使い回し | 類語辞典的な言い換えでなく、より具体的な語（固有名詞・数値・感覚的な描写）に置き換えられないか探す |
| `nested_attributive` | 連体修飾の入れ子 | `translationese.md` の該当パターンの通り、文を分割する |
| `english_syntax_inanimate_subject` / `inanimate_subject_morph` | 無生物主語+他動詞 | 主語を人や状況に戻すか、述語を状態描写に変える |
| `english_syntax_cleft_because` | 「それは〜。なぜなら〜」型 | 理由を先に書くか、1文にまとめる |

すべてのfindingsを機械的に潰す必要はない。検出は疑いの提示にすぎず、文脈上そのままで自然な箇所も混じる。判断に迷ったら、その一文を声に出して読んでみて、実際に人が話すときの言い回しに近いかどうかで決める。

## 素材不足の分岐

lint のどのカテゴリにも引っかからないのに、段落を読み返すと一般論しか言えていないことがある。固有名詞がなく、数値がなく、実例もない。誰が書いても同じ内容になる段落は、文体を整えても中身が変わらない。原因は書き方でなく素材の不足にある。現行の lint はこの種の空疎さを検出できない（`low_specificity` のような専用カテゴリは今のところ実装されていない）。表層の禁止語や統語パターンとは別種のAI臭であり、修正パスの中で見つけたら自己点検で拾うしかない。

見分け方は単純で、段落から固有名詞・数値・実例をすべて抜いても文意が変わらないなら、それは一般論止まりのサインになる。この状態を見つけたときは、言い回しを変えるループを回し続けても解決しない。手を止めて「素材集め（再帰的推論リサーチ）」の節に戻り、その段落を支える固有名詞・数値・一次情報を集め直してから書き直す。

## 判断台帳

lint の finding は疑いの提示であって指示ではないぶん、「見た・考えた・流した」のどこかで判断が宙に浮きやすい。それを防ぐため、finding ごとに「直した」か「残す（理由）」かをその場で記録しながら進める。

台帳の置き場所は次の優先順位で選ぶ。

1. **第一候補: チャットの応答テキストの中で管理する。** ファイルを作らずに済むならそれに越したことはない。1周あたりの finding 数が扱える範囲なら、応答の中に台帳を書きながら進めれば追跡できる
2. **次善策: 文書が長い・周回数が多いなどでチャット内管理が煩雑になった場合に限り、ファイル化する。** その場合も対象文書と同じディレクトリには置かない。環境のスクラッチパッド（用意されていればそこ）、なければ `mktemp -d` で作った一時ディレクトリに置く

どちらの置き場所でも、台帳はユーザーのプロジェクトに残すものではなく、作業の途中経過を追うための一時的なメモである。ファイル化した場合の後片付けは「作業ファイルの扱い」を参照。

最低限、次の形で1行ずつ書く。

```
- [直した] paragraph_lead_conjunction: 3段落目冒頭「しかし、」を削除し前段落と地続きにした
- [残す/固有名詞由来] forbidden_phrase「多角的」: 製品名「多角的レビュー機能」の一部なので変更不可
- [残す/文脈上必要] antithesis_repetition: この段落の対比は論旨の核なので残す
- [残す/技術用語] translationese「〜することができる」: API仕様書の可能表現として正確さを優先
```

「残す」を選ぶときは、理由の種類を一言添える（固有名詞由来・文脈上必要・技術用語・ジャンル上自然、など）。理由なしの「残す」は判断を放棄しているのと同じなので避ける。

台帳は周回のたびに追記していく。前の周で「残す」と決めた項目は、次の周で再度悩み直す必要はない。

周回のたびに全 finding を目視で見比べる代わりに、前回の `--json` 出力を `ai-smell-lint.py --baseline` に渡すと、今回の finding を resolved/new/persisting に自動仕分けできる。台帳への追記は new と persisting だけを見れば足りる。

## 発散ガード

同じ finding が2周連続で「直しては再発」する場合、その finding にこれ以上機械的な修正を重ねても収束しない。次のどちらかで抜ける。

- 台帳上「残す」に倒し、理由を「この形の反復自体がこの文書の文体として必要」などと明記する
- 文単位の言い換えではなく、その文・段落の構造そのものを書き直す（語順の入れ替え、文の分割・統合、視点の変更など）。書き直したら新しい finding として扱い、台帳の該当行を更新する

これは回数で機械的に打ち切るためのルールではない。「同じ箇所が同じ形で再発する」という状態そのものが、その finding がもう単発修正では扱えないという合図であり、それを検知したら状態を切り替える、という遷移ルールである。

## 自己点検ループ

下書きを直したあと、一度時間を置くか視点を切り替えて読み直す。おすすめの型は「これを他人が読んだら、どこに引っかかるか」を具体的に想像しながら読むことで、次の観点を順に当てる。

1. 冒頭の3段落だけを読んで、結論を先取りしすぎていないか
2. 段落ごとの文末を目で追い、同じ形（体言止め・「〜だ」・「〜である」）が連続していないか
3. 声に出して読んだときに息継ぎの位置が不自然でないか（長すぎる一文の兆候）
4. 自分が本当に思ったことと、書かれている内容がずれていないか。借り物の意見に見える箇所はないか

このループで気づいた点があれば台帳に新しい行として起こし、lint の再検知ループに戻る。何も引っかからなくなったら、そこがゴールになる。

## 収束の状態

このスキルのループは回数ではなく状態で終わる。出口は次の2条件が両方そろったときだけ。

1. 台帳上、すべての finding が「直した」か「残す（理由つき）」のどちらかに仕分けられている
2. 「直した」による修正が、新しい finding を生んでいない（lint を再実行しても新規発生がない）

この2条件がそろって初めて lint 側の収束であり、そこから最終パス（自己点検ループ）に進む。最終パスで違和感が出た場合は、それを台帳に新しい項目として追加し、収束条件を再度満たすまでループに戻る。

同じカテゴリの finding がなかなか減らないときは、次のどちらかを疑う。

- 閾値の問題: その文書のジャンル・文体では、指摘された特徴がむしろ自然なことがある（対談調のエッセイで体言止めが多いなど）。その場合は台帳に理由を書いて「残す」に倒す
- 構造の問題: 単発の修正では直らない、文章全体の構成そのものに起因する反復である。段落の並べ替えなど、より大きな手直しが必要になる。それでも直らず同じ形で再発するなら「発散ガード」に従う

## 作業ファイルの扱い

このスキルの作業中は、判断台帳・`ai-smell-lint.py --json` の出力・下書きの途中版バックアップなど、いくつもの中間ファイルが生まれうる。これらはユーザーのプロジェクトに残してよいものではない。

- 中間ファイルを作る必要が生じたら、対象文書と同じディレクトリには置かない。環境のスクラッチパッド（利用可能ならそこ）、なければ `mktemp -d` で作った一時ディレクトリに置く
- ループの途中は、その一時ディレクトリの中で自由に読み書きしてよい。周回のたびに上書き・追記して構わない
- ループが収束し、最終パス（自己点検ループ）も通過して完了となった時点で、後片付けを行う。一時ディレクトリごと削除し、台帳ファイルや lint の JSON 出力をユーザーのプロジェクトはもちろん、一時領域にも残さない
- 最終的にユーザーのプロジェクトに残すのは、完成した文書本体と、ユーザーが明示的に望んだ場合の `style-profile.md` だけにする

台帳をチャット内で管理した場合（第一候補の運用）は、そもそもファイルが生まれないので後片付けの対象もない。ファイル化したときだけ、この後片付けを忘れずに行う。
